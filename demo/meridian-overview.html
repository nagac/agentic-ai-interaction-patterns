01 · Background

30 designers.
30 different AI experiences.

Designers across 30 products were building AI features independently with no shared design language. Users moving between products encountered different visual treatments, different interaction patterns, and different trust signals for identical AI behaviour.

The deeper problem is that standard UX design patterns were built for interactions where the user is present and in control at every step. Agentic AI breaks this assumption entirely.

The number that started everything
30
Designers building AI features independently across 30 products with no shared foundation.
Four challenges standard patterns cannot address
👁️
Visibility
Agents run without the user watching. How do they know what is happening across 40 simultaneous tasks?
⚖️
Trust at scale
A wrong agent decision on 100 products causes 100× the damage of a wrong human decision on 1.
⏸
Interruption
The user needs to stop agents mid-run. Standard cancel buttons were not designed for chains with partial completion.
👋
Re-entry
The user left and came back. The world changed without them. Standard dashboards have no concept of this.
02 · Design Principles

Five principles that
govern every decision.

When two design options are in conflict — apply the principle with the lower number. These are not guidelines. They are constraints.

01
Preserve completed work — always
An agent failure never rolls back the work of prior agents. Completed outputs are always kept regardless of what happens next in the chain.
02
Transparency before autonomy
Users must see what agents will do before they start. No exceptions. The approve click is the explicit consent moment.
03
One question at a time
When agents need the user — one specific question. Never multiple decisions at once. Cognitive load at decision moments must be minimal.
04
Urgency first
What needs the user is always shown before what succeeded. Good news waits. Issues are never buried below completions.
05
Contract over surprise
Every agent boundary was declared upfront in Pre-flight Plan. The system honours exactly what was agreed. Users don't fear AI doing things — they fear AI doing things they didn't expect.
03 · The 8 Patterns

One scenario.
Eight design problems.

All 8 patterns were derived from a single enterprise retail scenario: a user types one prompt that triggers 4 agents across 10 products simultaneously. Each pattern was tested against one question: does this problem only exist because AI acts autonomously?

The anchor scenario
"Generate videos, write descriptions, run QA and publish for all Summer Collection products"
Triggers 4 agents across 10 products simultaneously. 40 tasks. Running autonomously. This one scenario exposed every design challenge in the library.

Does this problem only exist because AI acts autonomously?
If the answer is no — the pattern was rejected. Standard UX patterns handle the rest. Only genuinely agentic problems are documented here.

04 · Agent State Language

Five states.
Used everywhere. Never changed.

The universal visual language shared across all 8 patterns and every product in the organisation. These are system tokens — not product choices. Designers cannot change these colors.

Running
#3B82F6 · Blue
Agent is actively working on this item right now. Progress is being made. No action needed from the user.
No action needed
Done
#10B981 · Green
Agent completed this stage successfully. Output is ready. Chain can proceed to next agent.
No action needed
Stuck
#EF4444 · Red
Agent hit a problem it cannot resolve alone. Always surfaces at the top. Always has a Fix button. Never hidden.
Fix required now
Waiting
#F59E0B · Amber
Agent paused at a pre-declared boundary. Awaiting user approval to continue. Not an error — an agreement being honoured.
Decision needed
Pending
#9CA3AF · Grey
Stage has not started yet. Waiting on a prior agent in the chain to complete before this one can begin.
No action needed
⚠ Critical rule — These five states and their exact color tokens must never change across any product in the organisation. Treat them as immutable system tokens.
05 · Container Hierarchy

Dialog. Sheet. Popover.
A three-level system.

The Pre-flight Plan decision established the container hierarchy for all primary decision moments in the library. Two questions determine the right container: Is the user in the middle of something? And how much space does the decision need?

Level
Container
Purpose

Primary
Dialog
Core consent and decision moments. Always centred. User must engage before anything starts. Cannot be ignored. The contract between user and agents.

Detail on demand. Slides up from the primary Dialog without replacing it. User can drill into agent workflow, full parameter list, or supporting information. Optional — not required to proceed.

Tertiary
Popover
Inline micro-edits. Appears next to the field being edited. Small, contextual, dismisses when done. Never stacks on top of another Popover.
Editing a single parameter value. Updating one field before starting a run. Quick contextual actions that do not warrant a full Sheet.
💡
Why not a Side Panel for Pre-flight?
A side panel forces users to shift focus to the edge of the screen. On wide monitors this means turning their head. If agents are run frequently, this becomes genuinely disruptive. Dialog is always centred. Users do not have to move — the decision comes to them. The prose concern is solved by the Sheet, not by changing the primary container.

06 · Key Design Decisions

Decisions made.
Reasoning documented.

Every significant design decision made during the development of this library is logged here. Not just what was decided — but why, and what alternatives were considered.

Decision
P-001 Container
Pre-flight Plan uses Dialog not Side Panel
Side panel forces users on wide screens to shift focus to the edge of the screen and turn their head. For frequent users running agents regularly, this becomes disruptive. Dialog is always centred — the decision comes to the user, not the other way around. Prose concerns solved by Sheet on demand.
Outcome
Dialog (primary) + Sheet (detail) + Popover (inline edits)
Decision
P-000 Input field
Instruction shown in Pre-flight as display element not disabled input
A disabled input signals broken or locked — not confirmed. Disabled form elements are also skipped by screen readers, which means users relying on assistive technology would miss seeing their instruction reflected back to them. A read-only text block or callout is the correct component for displaying confirmed information.
Outcome
Read-only display component. Not a form element.
Decision
Users vs Steps
Users see agents, not internal steps
Agents can have many internal steps and those steps can get very complex. That complexity is real but it belongs to engineering, not to the user. The patterns operate at the agent level. Internal steps stay invisible unless the agent genuinely needs a human decision. This keeps the interface manageable no matter how complex the underlying system becomes.
Outcome
Patterns surface agents. Steps are invisible by default.
Decision
P-007 Visual tone
Decision Checkpoint uses teal, not red
The Publishing Agent stopping at the checkpoint is not a failure. It is the agentic contract being honoured exactly as agreed in Pre-flight Plan. Red would communicate failure or urgency. Teal communicates a planned, positive handover moment. The agent worked correctly — it stopped when it was supposed to.
Outcome
Teal #0D9488 throughout P-007. Never red.
Decision
P-003 vs P-004
Graceful Handoff and Error Recovery are distinct patterns
The test: did the agent choose to stop (P-003) or was it forced to stop (P-004)? Graceful Handoff is amber — agent is working correctly and recognising its own limits. Error Recovery is red for user-caused issues and purple for system failures. Never use the same visual treatment for both.
Outcome
Two separate patterns. Different tone. Different color.
Decision
P-000 Evolution
Instruction Input supports 8 input types including Meta
As prompting becomes more sophisticated, a single text box is insufficient. The pattern now supports Text, File, Template, History, Form, Voice, URL, and Meta. Meta supports goal-driven, block builder, and context assembler modes. The clarity check always runs regardless of input type.
Outcome
8 input types. Meta is the 8th. Clarity check unchanged.
07 · How to Use This Library

Different audiences.
Different entry points.

Meridian is designed to be used by designers, engineers, and product leaders. Each audience has a different entry point and a different way of working with the library.

🎨
For designers
Building new features
1
Start with the design test — does this problem only exist because AI acts autonomously?
2
Find the pattern that matches your problem in the overview table.
3
Read the full pattern documentation — anatomy, variants, states, rules.
4
Use the Figma Make components. Do not recreate from scratch.
5
Check the usage guidelines — when to use, when not to use, how to combine.
⚙️
For engineers
Building the patterns
1
Read the anatomy table for every element and why it exists.
2
Implement all states documented in Section 04. Missing states become bugs.
3
Use the exact agent state color tokens. Never change them.
4
Implement the content guidelines — exact copy standards are documented.
5
Reference the Figma Make build guide for component structure and token mapping.
📊
For product leaders
Making decisions
1
Start here — this overview document covers everything at the right level.
2
Review the five design principles. These are the constraints the library operates within.
3
Check the key decisions log before proposing changes to existing patterns.
4
Use the metrics table in each pattern to measure whether patterns are working.
5
Refer new AI feature requests to the design test before building.
The four hard rules for all audiences
✕Never change the 5 agent state colors. They are system tokens.
✕Never skip Pre-flight Plan before any multi-agent task starts.
✕Never roll back completed agent work on a failure.
✕Never auto-publish without explicit user approval at Decision Checkpoint.
