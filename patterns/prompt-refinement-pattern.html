Prompt Refinement
How do you help the user write a prompt that agents can act on correctly — before they submit?
Real-time guidance that scores the instruction, surfaces what is missing, and suggests additions — so the user gets it right before Pre-flight loads.

Why this pattern exists
The clarity check in P-000 catches ambiguity after the user submits. But by then they have already committed to an instruction. Prompt Refinement catches it earlier — while they are still writing. The earlier ambiguity is caught, the less it costs.
Anatomy — 3 elements inline
1
Prompt quality scoreA live completeness bar showing how actionable the instruction is. Updates 500ms after the user stops typing. Green when ready, amber when missing elements, red when too vague.
2
Missing elementsSpecific things the instruction still needs. Plain language, never technical. Each one is a gap an agent would struggle with.
3
Suggested additionsTap-to-add chips that append the missing element directly into the instruction. User does not need to retype — one tap fills the gap.
When to use
✓ First-time users
Users who have never written an agent instruction before need active guidance to understand what makes a good prompt.
✓ High-scale runs
When agents will act on 50+ items, the cost of a vague instruction multiplies. Refinement is especially valuable here.
✓ Complex agent chains
4+ agents in sequence means more parameters to specify. Refinement surfaces what each agent will need.
When NOT to use
✗ Power users, familiar task
Returning users running a known task type find refinement guidance patronising. Make it dismissible or off by default for them.
✗ Template or form input
If the user selected a template or used the structured form, the instruction is already well-formed. Do not run refinement on these.
Hard stops
✕Never block submit — refinement is guidance, not a gate
✕Never rewrite the user's instruction — only suggest additions
✕Never show refinement for voice, file, URL, or form input types
✕Never show more than 3 missing elements at once — prioritise
✕Never keep showing refinement after user dismisses it
Relationship to other patterns
P-000 — Prompt Refinement lives inside P-000. It is a sub-pattern of Instruction Input, not a standalone screen.
Clarity check (P-000) — Refinement does not replace the clarity check. It reduces how often the clarity check needs to flag something. Both always run.
P-001 Pre-flight Plan — A well-refined instruction produces a Pre-flight Plan the user recognises and approves quickly. Fewer surprises, higher approval rates.
