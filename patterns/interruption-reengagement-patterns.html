P-005

Interruption & Pause

The user decides to stop agents mid-run — not because something broke, not because an agent asked. The user chose to intervene. The system must stop cleanly, preserve all work, and hold state indefinitely until the user is ready.

00 — Why This Is Agentic
The user is taking back control of an autonomous system
In standard UX, the user is always in control. There's nothing to interrupt. In agentic systems, agents are running autonomously — the user has to actively reach in and stop them. That requires a designed safe stop mechanism.
The agentic problem
Agents are running on 10 products simultaneously. The user spots something wrong — a wrong parameter, a changed brief, a business decision. They need to stop everything without losing what's already done.
The design challenge
Stop agents cleanly — not mid-output. Preserve all completed work. Hold state exactly. Give the user full visibility of where everything paused. Let them resume, adjust, or cancel.
The design decision
Agents finish their current product task then pause. Mid-task stop corrupts output. Clean stop preserves every completed product and resumes from the next pending one.
01 — The Pause Trigger
Where the pause button lives and how it behaves
The pause button must always be visible while agents are running. Never hidden, never buried. One click — confirm — agents begin clean stop.

Why this state exists
The user clicked pause but agents don't stop instantly. Each agent is mid-task. If we cut immediately, output is corrupted. This state tells the user: we heard you, we're stopping cleanly, here's which agents are still finishing.
⚠ Design rule
Never show a generic spinner. Always show which specific agents are finishing vs already paused. Users need to know exactly where the system is.

The three options explained
▶ Resume
Agents pick up from next pending task on each. No repeated work. Unchanged parameters unless user edited them in the paused state.
Edit → Resume
User changes a parameter in the paused state — QA threshold, tone, brand kit. Resume button becomes "Resume with updates". Updated parameters apply only to remaining products. Completed products are not re-run.
✕ Cancel run
All agent work stops permanently. Completed outputs are still preserved — nothing is deleted. User is taken to a summary of what was completed before cancellation.
04 — Rules
Do's and Don'ts — Interruption & Pause
✅ Always do this
✓Always show pause button while agents are running — never hidden
✓Always require confirmation before stopping — prevents accidental pauses
✓Always finish current task before pausing — never cut mid-output
✓Always show the finishing state — user must know the system heard them
✓Always show a snapshot of exactly where everything paused
✓Always preserve completed work — even if run is cancelled
❌ Never do this
✗Never cut agents mid-output — corrupts work and breaks trust
✗Never auto-resume — user must explicitly choose to resume
✗Never delete completed outputs when run is cancelled
✗Never show a generic spinner during finishing state
✗Never re-run completed products when resuming with updated parameters
✗Never hide the pause button inside a dropdown or menu
P-006

Re-engagement

The user left while agents were running. They come back — minutes, hours, or a day later. Agents have finished, failed, or changed state. The user needs to re-orient instantly — what happened, what needs them, what's done.

00 — Why This Is Agentic
The user was absent while the system acted autonomously
In standard UX the user is always present — there's no concept of "coming back." In agentic systems, agents run without the user watching. When the user returns, the world has changed without them. Re-engagement is the designed re-entry point.
The re-engagement problem — unique to agentic systems
🕐
Time has passed. The user has no mental model of what changed. They could have been gone 5 minutes or 5 hours.
🤖
Agents acted on their behalf without them watching. Some succeeded. Some failed. Some are waiting.
⚡
The user needs to re-orient instantly — what needs me right now — without reading through everything.
The design principle
Re-engagement is not a dashboard. It's a briefing. Like a good assistant telling you what happened while you were out — most important first, full detail available on demand, clear on what needs you and what doesn't.
01 — The Re-engagement Card
What the user sees when they return
A structured briefing. Four sections in fixed priority order: what needs you now, what's waiting for approval, what happened while you were gone, what's done. User can act directly from this card.

Priority order — always fixed
1st — Needs you now
Stuck agents, failures, critical errors. Something is blocked because of them. Always shown first.
2nd — Awaiting approval
Products ready to publish. Agents finished their work. Nothing is blocked — but user sign-off is needed before next step.
3rd — What happened
Timeline of events while user was away. Most recent first. Gives context without requiring the user to read everything.
4th — Completed
Everything that finished successfully. Shown last — user doesn't need to act on these. But they want to know.

03 — Rules
Do's and Don'ts — Re-engagement
✅ Always do this
✓Always show how long the user was away — gives temporal context
✓Always show issues first — never bury them below good news
✓Always show the summary strip — 4 numbers, instant orientation
✓Always show what changed in chronological order — newest first
✓Always provide a direct action CTA — never just information
✓Always use green treatment when everything went well — don't understate good outcomes
❌ Never do this
✗Never show completed items before issues — good news first is wrong here
✗Never dump a full log — the timeline shows key events only, not everything
✗Never require the user to navigate elsewhere to see what happened
✗Never show the same treatment regardless of what happened — match variant to outcome
✗Never omit the away time — "while you were away" without context is useless
✗Never show re-engagement card if agents are still running and nothing needs the user — use Variant C instead
04 — Metrics
How you know both patterns are working
P-005 · Time to resume after pause
<5min
From fully paused state to user clicking resume. Longer means the paused state is confusing.
🚨 Red flag if over 20 min
Users aren't confident resuming is safe
P-005 · Cancel rate after pause
<20%
How often users cancel entirely instead of resuming after pausing.
🚨 Red flag if over 50%
Users pausing because something is wrong — not just to adjust
P-006 · Time to first action on return
<30s
From re-engagement card appearing to user clicking their first action.
🚨 Red flag if over 2 min
Briefing not clear enough — user spending time reading not acting
