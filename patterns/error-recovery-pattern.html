Error Recovery

When an agent fails unexpectedly mid-chain — preserve all completed work, tell the user exactly what broke and why, then let them fix the cause and resume from the exact failure point.

00 — How This Differs From P-003
Graceful Handoff vs Error Recovery
These two patterns look similar. They are fundamentally different. Every designer using this library must know the distinction.
P-003 — Graceful Handoff
🧠Agent knew it needed help. It paused deliberately.
❓A decision is needed — not a fix. Agent is not broken.
▶After user decides — agent resumes immediately. No repair needed.
😐Tone: calm, collaborative. Agent asking for guidance.
P-004 — Error Recovery
💥Agent didn't know. It broke. Something failed unexpectedly.
🔧A fix is needed — not just a decision. Agent cannot resume as-is.
🔁After user fixes the cause — agent resumes from failure point. Not from scratch.
⚠️Tone: clear, direct. Something went wrong. Here's what. Here's how to fix it.
The test to use the right pattern
Ask: "Could the agent have continued if it wanted to?"
Yes — use Graceful Handoff. The agent chose to stop and ask.
No — use Error Recovery. The agent could not continue regardless.

03 — Error Card
What the user sees when they open a failed item
Three sections. What's safe. What failed and why. How to fix it and resume. Never show a raw error log. Always translate to plain language.
The error card
✕
Description Agent failed
Jordan 1 Retro · SKU-0078 · Stage 2 of 4
✓ Work preserved from earlier stages
🎬Video Agent output saved — 30-second video ready
✕ What failed
❌Description Agent could not access the product data file. The file was moved or renamed since the run started.
⊘ Blocked until fixed
—QA Agent — waiting on description
—Publish Agent — waiting on QA
Root cause
Product data file not found
Expected: /products/jordan1_data.json
File was moved or renamed after the run started.
How to fix and resume
1
Restore or rename the product data file to its original location.→ Open file manager
2
Confirm the file is accessible, then resume the Description Agent from this point.
The three non-negotiable sections
1 — What's safe
Always shown first. Users fear losing everything. Immediately reassure them about what work was preserved before explaining what failed.
→ Never skip this section
2 — What failed and why
Plain language. Never raw error codes. Translate the technical cause into what the user can actually understand and act on. Always include what's now blocked downstream.
→ Never show raw stack traces here
3 — How to fix and resume
Numbered steps. Specific actions. Not "try again" — exactly what to do. Resume button picks up from failure point, never from scratch.
→ Resume always means from failure point

05 — Rules
Do's and Don'ts
These rules protect user trust when the system breaks. Trust is hardest to rebuild after unexpected failures.
✅ Always do this
✓Always show what's safe and preserved before showing what failed
✓Always translate technical errors into plain language the user understands
✓Always show what's blocked downstream — not just what failed
✓Always resume from the failure point — never restart from scratch
✓Always distinguish system errors (purple) from user-caused errors (red)
✓Always give numbered fix steps — never just "try again"
❌ Never do this
✗Never show raw error codes or stack traces in the error card
✗Never roll back completed work from prior agents
✗Never use urgent notification for a single product failure
✗Never imply user error when it was a system failure
✗Never offer only "Start over" — always offer "Resume from here"
✗Never silently retry without telling the user
